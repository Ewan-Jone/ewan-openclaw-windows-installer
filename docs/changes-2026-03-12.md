# 2026-03-12 凌晨改动总结

## 背景

同事机器安装 1.2.6 时，check.bat exit 1，但 `D:\OpenClaw\logs\` 下没有 check 日志，只有 install 日志记录了一行 `check.bat exit: 1`。排查发现是 Inno Setup 安装步骤顺序的根本性误解。

---

## 核心发现：Inno Setup 安装步骤顺序

Inno Setup 的 `CurStepChanged` 回调有两个关键步骤：

| 步骤 | 含义 |
|------|------|
| `ssInstall` | 文件拷贝**之前**触发，此时 `{app}` 目录为空 |
| `ssPostInstall` | 文件拷贝**之后**触发，此时所有文件已在 `{app}` |

之前的代码在 `ssInstall` 里执行 check.bat，但这时 `{app}` 根本没有 check.bat，所以脚本找不到、执行失败、exit 1，日志也没有（文件不存在无法写入）。

---

## 改动内容

### 1. `scripts/setup_checks.pas`

**之前：** check.bat 和 install.bat 都在 `ssInstall` 里执行。

**之后：** 两个脚本都移到 `ssPostInstall` 里执行，流程如下：

```
ssPostInstall 触发（文件已拷贝到 {app}）
  ↓
ForceDirectories({app}\logs)   ← 先建日志目录
  ↓
运行 {app}\check.bat           ← 环境检查，日志写到 {app}\logs\openclaw-check.log
  ↓ 失败则 Abort（中止整个安装，不可继续）
运行 {app}\install.bat         ← WSL 安装，日志写到 {app}\logs\openclaw-wsl-install.log
  ↓ 失败则 Abort
SaveLog()                       ← 把所有日志复制到桌面 openclaw-logs\
```

**关键点：**
- 失败时用 `Abort` 而不是 `WizardForm.Close`，确保安装彻底中止，用户无法继续
- check.bat 被调用后第一行就写日志，可以区分"没被调用"和"调用了但失败"

### 2. `openclaw-launcher/src/wsl.rs`

`start_gateway` 函数签名改动：端口从写死 17789 改为参数传入。

```rust
// 之前
pub fn start_gateway(distro: &str) -> Result<()>

// 之后
pub fn start_gateway(distro: &str, port: u16) -> Result<()>
```

所有调用处更新：
- `main.rs` 后台启动线程：`start_gateway(&distro, gateway_port)`
- `main.rs` 托盘重启按钮：`start_gateway(&distro, port)`
- `main.rs` onboarding 配置后重启：`start_gateway(distro, oc.webchat_port)`
- `health.rs` 健康检查重启：`start_gateway(distro, 17789)`（暂时写死）

### 3. `openclaw-setup.iss`

去掉了临时添加的 check.bat 复制到 `{tmp}` 的条目（中间调试版本的产物，已清理）。
